## 6. Functional-bug findings — false-positive sweep

The QA-lens agent flagged 6 "critical" items. After reading the actual
code:

| Finding | Verdict |
|---|---|
| `requiredTasks.js:245` `Promise.all` not awaited | FALSE POSITIVE. Each inner `addDoc(...).catch(...)` already swallows individual failures into a `console.warn`. Outer Promise.all returns and the function returns the campaign result. Audit writes are intentionally fire-and-forget so a logging hiccup doesn't block the campaign. Comment at line 243-244 explains this. |
| `requiredTasks.js:271,295,320` fire-and-forget audit writes | FALSE POSITIVE. Same pattern — each has `.catch(() => {})` or `.catch(e => console.warn(...))`. Intentional. |
| `coverage.js:94-103` Promise.all swallows failures | FALSE POSITIVE. Each `notifyStaff(...).catch(() => {})` wraps individual sends. Outer loop iterates without Promise.all (it's a `for...of`), so one failure cannot reject anything. |
| `OnboardingPortal.jsx:145` `updateDoc` fire-and-forget | INTENTIONAL. Comment at 143 says "Best-effort mark invite as opened (non-blocking)." A failure here legitimately should not block the hire from loading their portal. `.catch(() => {})` is correct. |
| `Schedule.jsx:2450-2460` `Promise.all` on FCM fan-out | FALSE POSITIVE. Each `notify(...).catch(() => {})` is per-recipient. Promise.all over already-caught promises cannot reject. |
| `OnboardingApply.jsx:1078` regex `/^image\/\|^application\/pdf$/` | FALSE POSITIVE. The `\/` is just an escaped forward slash (cosmetic — JS regex doesn't need slashes escaped, but it's legal). The `|` is alternation, not a literal. Regex matches `image/*` OR `application/pdf` correctly. Tested: `image/png` → true, `application/pdf` → true, `text/plain` → false. |

**Net new functional bugs: zero.** Six false positives. The codebase
has a consistent `.catch` discipline that the auditor missed because
it scanned for `Promise.all` patterns without reading the inner
calls. This is good news.

## 9. Remaining issues — owner review

Andrew should ack these before we wrap. None are deal-breakers.

1. **Print-on-iOS uncertainty.** AirPrint from a Capacitor WebView is not as reliable as Safari. Schedule print, InventoryHistory print, PrepList print all call `window.print()`. If real-device test shows broken behavior, fallback options:
   - Generate the print HTML, export to PDF via `pdf-lib`, save via `@capacitor/filesystem`, share via `@capacitor/share`.
   - Or open the print URL in `@capacitor/browser` (system Safari) which has reliable AirPrint.
2. **Account deletion UX.** Spec is in `CAPACITOR_PREP.md` Part 9. Recommend: button in My Profile → glass confirm modal → Cloud Function writes a deletion request doc → admin approves in AdminPanel within 7 days → soft-archive (keep audit log, remove from staff list, wipe FCM tokens, mark hire as archived). Acceptable to ship the request-side first, automate admin side later.
3. **Phase 2 Firebase Auth + App Check.** Documented for a future quarter. Not blocking launch; the existing client-gate + narrow rules + audit log posture is acceptable for an internal-only staff app.
